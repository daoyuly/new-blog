# Loop Engineering：以诗词网站为例的自运营系统设计

## 什么是 Loop Engineering

Loop Engineering 是一种面向高级目标的完成机制。它不把系统当作一次性交付的产物，而是当作一个**持续运转、自我优化的闭环**。核心思路：

1. **系统即运营** — 不存在"开发完"这个节点，系统上线只是起点
2. **指标驱动迭代** — 用真实数据（阅读量、点赞量）而非直觉驱动每次改进
3. **工具即回路** — 分析工具、选择器、生成器都是回路上的齿轮，自动衔接

用一张图理解：

```
用户反馈 → 数据分析 → 洞察 → 调整策略 → 内容生成 → 用户触达 → 反馈...
     ↑                                                          ↓
     └────────────── Loop Engineering ──────────────────────────┘
```

这不是传统的"开发→上线→维护"线性模式，而是一个永远在转的飞轮。

## 具体实例：每日诗词网站

### 产品定义

一个诗词展示网站，核心交互极简：**每天展示一首诗**。

但"选哪首诗"这件事，藏着 Loop Engineering 的全部精髓。

### 诗从哪里来：多维度选择器

诗词选择不是随机的，也不是编辑人工定的。它由一个**诗词选择器**自动完成，输入维度包括：

| 维度 | 数据来源 | 示例 |
|------|---------|------|
| 时间 | 系统时钟 | 清晨→王维《山居秋暝》；深夜→李白《月下独酌》 |
| 地点 | IP/定位 | 北京→《燕歌行》；杭州→《钱塘湖春行》 |
| 天气 | 天气API | 下雨→《春夜喜雨》；大风→《茅屋为秋风所破歌》 |
| 用户输入 | 文字/标签 | 用户说"思乡"→《静夜思》 |
| 用户记忆 | 历史交互 | 已展示过的降权；用户点赞过的风格提权 |
| 历史上的今天 | 历史数据 | 6月15日→与该日期相关的诗词事件/诗人诞辰 |

**选择器算法**本质上是一个加权评分模型：

```python
score = w1 * time_relevance + w2 * location_relevance + \
        w3 * weather_relevance + w4 * user_intent + \
        w5 * novelty + w6 * historical_relevance
```

权重 `w1...w6` 不是固定的——它们由 Loop Engineering 的迭代机制自动调整。

### 怎么展示：记忆规律驱动的缺字/消失展示

这可能是最有趣的部分。展示一首诗不是一次性全部显示，而是模拟**人类记忆的规律**：

1. **首次展示**：完整显示全诗，伴随赏析
2. **二次遇见**：随机缺1-2个字，用户需脑补（激活回忆）
3. **三次遇见**：缺字更多，或诗句逐行淡出消失
4. **四次及以上**：只显示诗名或首句，用户需回忆全诗

缺字规则设计：

```
展示次数 → 缺字比例
  1次    →   0%
  2次    →  15%
  3次    →  35%
  4次    →  55%
  5次+   →  70%（只保留诗名+作者）
```

消失效果：诗句从末尾开始渐隐，用户可以点击"我想起来了"展开，也可以选择"提示"显示首字。

这背后的认知科学依据：

- **间隔重复（Spaced Repetition）**：艾宾浩斯遗忘曲线——在即将遗忘时复习，记忆最牢固
- **提取练习效应（Retrieval Practice Effect）**：主动回忆比被动重读更有效
- **必要难度（Desirable Difficulties）**：适度增加回忆难度，长期记忆效果更好

所以用户在这个网站上不是"读诗"，而是**在记住诗**。

### 迭代指标：阅读量与点赞量

Loop Engineering 的灵魂在于：**用数据验证假设，用迭代逼近目标**。

#### 核心指标

| 指标 | 定义 | 优化方向 |
|------|------|---------|
| 日阅读量 | 每天打开页面的独立用户数 | 选诗精准度、推送时机 |
| 点赞率 | 点赞数 / 阅读量 | 诗词质量、展示体验、缺字策略 |
| 回访率 | 7天内重复访问的用户比例 | 记忆机制粘性、内容新鲜感 |
| 完忆率 | 缺字展示时用户点击"我想起来了"的比例 | 缺字难度是否合适 |

#### 指标驱动迭代的实例

**假设**：用户更喜欢与天气相关的诗。

**验证**：对比晴天和雨天，天气匹配诗的点赞率是否更高。

**结果**：雨天匹配率 +23%，晴天无显著差异。

**行动**：雨天加大天气维度权重 `w3`；晴天降低 `w3`，提升时间维度权重 `w1`。

这就是一个完整的 Loop：

```
观察数据 → 形成假设 → A/B测试 → 验证/推翻 → 调整参数 → 观察数据...
```

### 工具链：回路的齿轮

Loop Engineering 需要工具把每个环节串联起来，形成自动化回路：

```
┌─────────────────────────────────────────────────┐
│                 Loop Engineering                 │
│                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐    │
│  │ 数据分析  │──→│  洞察生成 │──→│ 策略调整  │    │
│  │  工具    │   │   工具   │   │   工具   │    │
│  └──────────┘   └──────────┘   └──────────┘    │
│       ↑                              │          │
│       │         ┌──────────┐         │          │
│       └─────────│ 内容生成  │←────────┘          │
│                 │  工具    │                    │
│                 └──────────┘                    │
│                      │                          │
│                      ↓                          │
│              ┌──────────────┐                   │
│              │  诗词选择器   │                   │
│              │  展示渲染器   │                   │
│              │  用户交互层   │                   │
│              └──────────────┘                   │
└─────────────────────────────────────────────────┘
```

#### 工具1：数据分析与洞察

- **数据采集**：页面访问、停留时长、点赞/踩、缺字展开率、回访频次
- **聚合分析**：按维度（天气/时间/地域/诗词类型）拆解指标
- **异常检测**：某天阅读量骤降 → 自动排查是否选诗不当
- **洞察输出**：每日生成数据简报，推送至运营者

```json
{
  "date": "2026-06-14",
  "insights": [
    "雨天+思乡类诗词点赞率比均值高 31%",
    "缺字率 35% 时完忆率最高（72%），55% 时骤降至 41%",
    "深夜时段（22:00-01:00）用户停留时长是白天的 2.3 倍"
  ],
  "actions": [
    "将缺字第三档从 55% 调整为 45%",
    "深夜时段优先推送意境深沉的诗词",
    "雨天提高思乡/写景诗的权重"
  ]
}
```

#### 工具2：诗词选择器

- **诗词库**：唐诗三百首 + 宋词三百首 + 诗经选 + 用户投稿
- **标签体系**：每首诗标注主题（思乡/边塞/田园/爱情）、情绪（豪放/婉约/沉郁）、时令（春/夏/秋/冬/节日）
- **动态权重**：权重由数据分析工具的输出自动调整
- **去重策略**：已展示的诗词按遗忘曲线降权，间隔 2/7/30 天后重新出现

#### 工具3：展示渲染器

- **缺字引擎**：根据展示次数计算缺字比例，随机选取缺字位置（避开标题和作者）
- **消失动画**：CSS 渐隐效果，从末句开始，3秒淡出
- **交互层**：点击展开、提示首字、分享卡片生成

## Loop Engineering 的完整运转示例

以一个完整周为例：

### Day 1（周一）

- **选择**：晴天·北京·上午 → 王维《鹿柴》
- **展示**：首次完整展示
- **数据**：阅读 1,200，点赞率 8%

### Day 2（周二）

- **选择**：阴天·杭州·下午 → 苏轼《饮湖上初晴后雨》
- **展示**：首次完整展示
- **洞察**：阴天+写景诗点赞率偏低（6%），低于均值
- **调整**：阴天时降低写景权重，提升情感类权重

### Day 3（周三）

- **选择**：雨天·上海·深夜 → 李商隐《夜雨寄北》
- **展示**：首次完整展示
- **数据**：阅读 1,500（深夜用户暴涨），点赞率 14%！
- **洞察**：雨天+思乡+深夜 = 黄金组合

### Day 4（周四）

- **选择**：晴天·北京·清晨 → 孟浩然《春晓》
- **展示**：首次完整展示
- **特殊**：历史上6月18日是孟浩然相关纪念日，加成展示
- **数据**：阅读 900，点赞率 9%

### Day 5（周五）

- **选择**：阴天·成都·下午 → 杜甫《春夜喜雨》
- **展示**：首次完整展示
- **同时**：Day 1 的《鹿柴》再次出现（间隔5天，二次展示）
- **展示**：二次展示，缺字 15%，用户完忆率 78%

### Day 6（周六）

- **选择**：晴天·杭州·傍晚 → 白居易《暮江吟》
- **展示**：首次完整展示
- **洞察**：傍晚+写景诗点赞率回升（11%），不同于阴天场景
- **调整**：区分晴天傍晚和阴天下午的写景诗权重

### Day 7（周日）

- **周报生成**：
  - 本周阅读量 8,400，环比 +12%
  - 点赞率均值 9.6%，最高 14%（雨天深夜思乡）
  - 完忆率 78%，缺字 15% 档位效果最佳
  - 自动调整权重：雨天 w3↑、阴天写景↓、深夜思乡↑
  - 下周重点测试：缺字 20% vs 15% 的完忆率差异

## 为什么这是 Loop Engineering 而不是普通的产品迭代

| 维度 | 传统产品迭代 | Loop Engineering |
|------|------------|------------------|
| 频率 | 周为单位，人工驱动 | 天/小时为单位，自动驱动 |
| 决策 | 产品经理拍脑袋 | 数据洞察→自动调整参数 |
| 工具 | 分散，靠人衔接 | 串联成回路，自动流转 |
| 目标 | 功能交付 | 指标提升 |
| 记忆 | 靠文档和会议 | 靠权重和模型参数 |
| 规模 | 受限于团队带宽 | 可无限扩展维度和场景 |

Loop Engineering 的本质是：**把产品经理的判断力编码为可自动执行的回路，让系统自己在运转中变好。**

## 技术架构建议

```
用户端（Next.js）
  ├── 诗词展示页（缺字渲染 + 消失动画）
  ├── 用户交互（点赞/展开/提示/分享）
  └── 数据上报（停留时长/交互事件）

API 层
  ├── /api/daily-poem     → 诗词选择器
  ├── /api/user-memory    → 用户记忆管理
  ├── /api/interaction    → 交互数据采集
  └── /api/insights       → 洞察推送

后端服务
  ├── 诗词选择器（多维度加权 + 动态权重）
  ├── 数据分析引擎（聚合 + 异常检测 + 洞察生成）
  ├── 记忆管理（遗忘曲线 + 间隔重复调度）
  └── 权重优化器（根据洞察自动调参）

数据层
  ├── 诗词库（PostgreSQL + 向量索引）
  ├── 用户行为（ClickHouse）
  └── 权重参数（Redis + 版本管理）
```

诗词库的向量索引可以支持语义匹配——用户说"想看写秋天的"，不只匹配"秋"字，还能匹配"霜""落叶""登高"等语义关联。

## 总结

Loop Engineering 不是技术框架，是一种**思维模式**：

1. **定义可量化的目标**（阅读量、点赞率、完忆率）
2. **构建自动运转的回路**（数据→洞察→调整→生成→触达→数据）
3. **让系统在运转中进化**（权重自调整、策略自优化）

诗词网站只是一个例子。同样的思路可以用于：每日一首歌、每日一段代码、每日一个单词——任何"每天一个"的场景，都可以用 Loop Engineering 让它越运转越好。

而缺字展示和消失展示的设计，让这个系统不只是在"推送内容"，而是在**帮助用户建立长期记忆**。这是 Loop Engineering 的更高目标：不是让用户消费更多，而是让用户真正获得更多。
